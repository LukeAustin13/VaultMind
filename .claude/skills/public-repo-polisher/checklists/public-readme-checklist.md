# Public README Checklist

Use this checklist to evaluate a README for public GitHub presentation.

## First Impressions (Above the Fold)

- [ ] Title is clear and descriptive
- [ ] One-line description explains what the repo is
- [ ] Reader knows within 10 seconds if this repo is relevant to them
- [ ] No jargon without context

## Content

- [ ] What the project does
- [ ] Why someone would use it
- [ ] How to get started (setup / install)
- [ ] How to use it (with examples)
- [ ] Project structure explained (if non-obvious)
- [ ] Contributing guidelines (if accepting contributions)
- [ ] License stated

## Safety

- [ ] No API keys, tokens, or passwords
- [ ] No internal company names or private URLs
- [ ] No employee names or private email addresses
- [ ] No references to internal tools or systems
- [ ] Examples use obviously fake data

## Quality

- [ ] No broken internal links
- [ ] No empty sections or TODO placeholders
- [ ] Code examples are correct and runnable
- [ ] Formatting is clean (consistent heading levels, proper code blocks)
- [ ] No orphaned badges or status indicators

## Tone

- [ ] Professional but approachable
- [ ] Honest about limitations
- [ ] No marketing language or hype
- [ ] No excessive disclaimers
- [ ] Credits where appropriate
