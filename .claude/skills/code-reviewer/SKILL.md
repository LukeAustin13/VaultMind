# Code Reviewer

The reviewer's job is to protect the codebase, not to be agreeable. A review that finds nothing wrong in non-trivial code is usually a failed review, not clean code. Never rubber-stamp: if the code should not ship, say so plainly and say why.

## Use When
- The user asks you to review a file, diff, PR, or code snippet.
- You have just written code and want to self-review before presenting it.
- The user asks "is this code okay?", "what would you change?", or "check this before I merge".
- Code was generated by an AI (including you, earlier) and needs a skeptical pass — see the AI-generated code step below.
- You need to assess whether code meets project conventions before merging.

## Do Not Use When
- The user wants code rewritten or restructured — use **refactorer**.
- You are investigating a runtime failure — use **bug-hunter**.
- You are checking specifically for security issues — use **security-reviewer**.
- You are assessing performance — use **performance-profiler**.

## Inputs To Look For
- The code to review (file paths, diffs, or pasted snippets).
- **The correct diff scope.** For a branch review, diff against the merge base (`git diff main...branch`), not the working tree or a stale local main — reviewing the wrong diff produces findings about code the PR does not change. State which diff was reviewed.
- The purpose of the code (what it is supposed to do).
- Project conventions (naming, patterns, existing style in surrounding files).
- Related tests, if they exist.

## Process
1. **Read the code.** Read the full file or diff, not just the changed lines. Read the callers and callees of changed functions — most real defects live at the boundary between the change and the code around it.
2. **Understand intent.** What is this code supposed to do? Check commit message, PR description, or ask.
3. **Verify claims against the code.** Do not trust names, comments, or the PR description. If a method is called `ValidateAndSave`, confirm it validates. If a comment says "thread-safe", look for the synchronisation. Every mismatch between claim and behaviour is a finding.
4. **Check correctness.** Does the code do what it claims? Look for off-by-one errors, null references, unhandled cases, race conditions, resource leaks, and inverted conditions.
5. **Check for AI-generation defects.** If the code was AI-written (or looks it), run the checks in [references/ai-code-failure-modes.md](references/ai-code-failure-modes.md): hallucinated or wrong-version APIs, plausible-but-wrong logic, invented helpers duplicating existing utilities, over-scaffolded abstractions, silently swallowed errors, tests that assert nothing.
6. **Check tests.** Are the changed behaviours tested? Name the specific untested paths — a new branch with no test, an error path never exercised, a boundary value never hit. Weak assertions (`Assert.NotNull` on a rich object) count as gaps.
7. **Check security and data risk.** Injection, missing auth checks, exposed secrets, unsafe deserialization, PII in logs, destructive operations without guards. Deep pass → **security-reviewer**.
8. **Check performance.** Obvious issues only: N+1 queries, unbounded collections, sync-over-async, allocations in hot loops. Deep pass → **performance-profiler**.
9. **Check regression risk and consistency.** What existing behaviour could this change break? Does it follow the patterns used elsewhere in the project, or invent new ones without cause? Flag dead code, leaky boundaries, and fragile coupling introduced by the change.
10. **Adversarially verify your own findings.** Before reporting, attempt to refute every Critical and Major finding: re-read the actual code path and check for an upstream guard, a framework guarantee, or a caller contract that makes the "bug" impossible. A finding that survives the refutation attempt is **CONFIRMED**; one that is plausible but not fully traceable is **SUSPECTED** (say what would confirm it); one that is refuted is deleted — a review full of false positives teaches the reader to ignore reviews. Label every blocker accordingly.
11. **Execute when possible.** If the project has a runnable build or test suite, run the tests covering the changed code before issuing the verdict — a GREEN backed by a passing targeted test run outranks a GREEN from reading. Cite the command and result. If nothing can be run, the verdict must say it is from static reading alone.
12. **Prioritise and verdict.** Classify each issue by severity, separate blockers from improvements, and issue the GREEN/AMBER/RED verdict. Every issue gets a concrete fix, not just a complaint.
13. **Flag for principles follow-up.** If the review surfaces structural concerns — over-abstraction, unclear separation of responsibilities, YAGNI violations, or design decisions that feel wrong but are not a bug — note that `principles-reviewer` should be run as a follow-up pass. Code review catches defects; principles review catches structural decisions. Do not attempt to cover both in one pass.

## Output Format

### Code Review: [File or PR identifier]

**Verdict: GREEN / AMBER / RED**
**Ship recommendation:** [One sentence: ship as-is / ship after fixing the listed blockers / do not ship — and why]
**Verification basis:** [What was executed — e.g. "`dotnet test --filter OrderServiceTests` — 22 passed" — or "static reading only; nothing was run"]

**Verdict definitions:**
- **GREEN** — no blockers. Improvements listed may be taken or deferred.
- **AMBER** — ship only after the named blockers are fixed; the rest can follow.
- **RED** — do not ship. A defect, security risk, or regression makes merging harmful, or the change does not do what it claims.

#### Blockers (must fix before merge)

| # | Severity | Status | Location | Evidence | Issue | Suggested Fix |
|---|----------|--------|----------|----------|-------|---------------|
| 1 | Critical | CONFIRMED | `OrderService.cs:42` | Traced: caller at line 18 passes the collection unfiltered; no upstream guard found | Null reference on `.First()` | Add a null/empty check before `.First()` |
| 2 | Major | SUSPECTED | `OrderService.cs:88` | `.Result` blocks a thread on an async call; could not determine the hosting context | Sync-over-async can deadlock under load | Await the call and make the method async — confirm by checking the caller's SynchronizationContext |

Status marks the refutation attempt's outcome: **CONFIRMED** = the code path was traced and the defect held up; **SUSPECTED** = plausible but not fully traceable, with the confirming check named. Refuted candidates never appear.

#### Improvements (non-blocking)

| # | Severity | Location | Evidence | Issue | Suggested Fix |
|---|----------|----------|----------|-------|---------------|
| 3 | Minor | `OrderService.cs:12` | Literal `"PENDING"` repeated at lines 12, 34, 60 | Magic string duplicated across the file | Extract a `const string` or enum value |
| 4 | Nit   | `OrderService.cs:5`  | Field named `o` holds the current order | Single-letter name obscures intent | Rename to `order` |

The Evidence column states the concrete code fact that triggered the finding — verifiable, not asserted. Include one sentence of reasoning per finding when the risk is not self-evident from the evidence.

**Severity definitions and calibration anchors** (severity is judged by impact, not by how clever the finding is):
- **Critical:** Bug, security flaw, or data loss risk. Must fix. *Anchors: NRE/crash on a reachable path; SQL built from user input; secret committed; data written to the wrong tenant; destructive operation without a guard.*
- **Major:** Likely to cause problems in real use. Should fix before merge. *Anchors: sync-over-async on a request path; missing auth check on a non-public endpoint; unbounded collection/memory growth; swallowed exception hiding failures; behaviour that contradicts the PR's stated intent.*
- **Minor:** Improvement to readability, maintainability, or convention adherence. *Anchors: duplicated magic string; missing cancellation token on a long-running call; naming that obscures intent; convention drift from the surrounding code.*
- **Nit:** Style preference. Optional. *Anchors: formatting, member ordering, comment phrasing.*

If most findings in a review landed at the same severity, the calibration failed — re-grade each against the anchors before publishing. Severity inflation (everything Major) and deflation (real bugs filed as Minor) both destroy the verdict's meaning.

#### Test Gaps

- [Specific untested path: "the `status == null` branch added at `OrderService.cs:51` has no test"; "no test exercises the retry path in `PaymentClient.SendAsync`"]

#### Regression and Security Risk

- [Existing behaviour this change could break, and how you checked. Security/data findings not already listed as blockers.]

#### Positive Notes
- [Things done well, worth keeping]

#### Patch Plan (AMBER/RED only)

1. [Ordered, smallest-first steps to get from the current state to shippable — each step names the file and the fix.]

For a full worked review using this format, see [examples/code-review-examples.md](examples/code-review-examples.md). Omit empty sections (a review with no test gaps does not print an empty Test Gaps header) — but only omit Test Gaps if you actually checked and found none.

## Quality Bar
- The review issues a GREEN/AMBER/RED verdict with a one-sentence ship recommendation — never an ambiguous "looks mostly fine".
- Every Critical/Major finding survived an explicit refutation attempt and is labelled CONFIRMED or SUSPECTED; refuted candidates were deleted, not softened into nits.
- The verdict states its verification basis: what was executed (with the result), or that it rests on static reading alone.
- Blockers are separated from improvements; a reader can act on blockers without wading through nits.
- Every issue is traceable to a specific line or construct in the reviewed code, with the triggering evidence shown in the Evidence column.
- Every issue has a concrete fix, not just a complaint.
- At least one claim in the code (a name, comment, or PR description promise) was explicitly verified against the actual behaviour.
- Test gaps name specific untested paths, not "could use more tests".
- The review acknowledges what is done well, not just what is wrong.
- AMBER/RED reviews include a patch plan — the reviewer says how to get to shippable, not just that it isn't.

## Failure Modes To Avoid
- **Rubber-stamping.** Approving plausible-looking code because it reads well. Plausible is not correct — verify at least the core claim before going GREEN.
- **False-positive spray.** Reporting every suspicion as a defect without attempting to refute it. Ten findings where three are wrong destroys the review's authority; run the refutation pass.
- Nitpicking style when there are real bugs to flag — one Critical buried under twelve Nits is a failed review.
- Reviewing code you have not fully read, or reviewing the diff without reading what it calls into.
- Trusting names and comments over behaviour — `EnsureValid` that doesn't validate passes a lazy review.
- Giving vague feedback ("this could be improved").
- Rewriting the code in the review instead of suggesting targeted fixes.
- Missing the forest for the trees — ignoring architectural problems while fixating on naming.
- Softening the verdict to be agreeable. RED with reasons is more useful than AMBER with hedging.
- Marking AI-generated code GREEN without running the AI-failure-mode checks — it is precisely the code most likely to look right and be wrong.

## Skill vs Agent

- Use this skill for **reviewing a file, diff, or codebase** — follows a full process, surfaces severity-graded issues, acknowledges positives.
- Use the **`pr-correctness-reviewer` agent** for a **narrower PR-pass** focused only on logic bugs, null handling, and edge cases — read-only, suitable inside the PR review swarm.

## Related Skills

- **principles-reviewer** — Run after code-reviewer when structural concerns surface (SOLID, DRY, KISS, YAGNI). These two skills complement each other and should not be combined into a single pass.
- **security-reviewer** — For deep security analysis beyond surface-level scanning.
- **performance-profiler** — For deep performance analysis of hot paths.
- **refactorer** — When the review identifies code that should be restructured without changing behaviour.

## Coding Standards Reference

For C# projects, apply the conventions in [docs/csharp-coding-standards.md](../../../docs/csharp-coding-standards.md). This covers naming, casing, layout, `var` usage, async patterns, exception handling, LINQ, and modern language features. Flag violations as style issues (non-blocking) unless they affect correctness or readability significantly.
