# Project Map Block

A project map is the one-page answer to "what is going on in this project?" — generated by
MindVault, readable by humans in Obsidian, and the cheapest orientation read an agent can
make. As of v0.7.0 the map is **not a separate note**: it is a generated block appended to
the end of the project hub note's body, between the `<!-- mindvault-generated:start -->` and
`<!-- mindvault-generated:end -->` markers. One page per project.

## Commands

```bash
mindvault map create --project "MindVault"     # add the map block to the hub (errors if present)
mindvault map rebuild --project "MindVault"    # refresh the generated block
mindvault map list
```

MCP: `mindvault_create_map`, `mindvault_rebuild_map`, `mindvault_list_maps`.

There is no `type: map` note any more — the block lives on the existing project note, so it
carries the hub's frontmatter and needs no folder or note type of its own.

## What the block contains

```markdown
# MindVault

(the human-written hub body: goal, non-negotiables, notes, links — untouched)

<!-- mindvault-generated:start -->
## Start Here
- route: `mindvault route --project "MindVault"`
- capsule: `mindvault capsule --project "MindVault" --mode coding`

## Key Decisions
- [[Decision - Use SQLite FTS5]] — accepted (2026-07-01)

## Active Tasks
## Open Risks
## Known Mistakes
## Do Not Repeat
## Constraints
## Work Areas
## Recent Sessions
## Recent Implementation Logs
## Reviews
## Related Prompts
## Needs Review
## Orphans
## Broken Links
## Large Notes Missing Summaries
## Organisation Score
## Last Rebuilt
2026-07-05 14:00 — `mindvault map rebuild --project "MindVault"`
<!-- mindvault-generated:end -->
```

The block deliberately **omits** the project's Current Goal, Non-Negotiables and a "Hub:"
pointer — that content already lives on the same page above the markers, so duplicating it
inside the block only created stale copies. What remains is navigation and health: routes,
the decisions/tasks/risks/mistakes in force, prevention rules, work areas, recent activity,
the hygiene flags (needs-review, orphans, broken links, large notes missing summaries) and
the organisation score line.

Every entry is a `[[wiki link]]` with status and date — compact refs, never note bodies.
Archived notes are excluded. Sections are bounded (10 entries for decisions/tasks/risks/
mistakes/constraints, 5 for logs/reviews/prompts). The Organisation Score line embeds only
the overall score and weakest category; run `organisation-score` for the full breakdown.

## The generated-block contract

- Everything between `<!-- mindvault-generated:start -->` and
  `<!-- mindvault-generated:end -->` belongs to MindVault and is rewritten on rebuild.
- **Everything outside the markers is human text and is preserved verbatim** — the whole
  hub body above the block is yours; write freely. A test proves rebuild keeps it.
- Rebuild snapshots the hub first, like every mutation.
- If the markers were deleted, rebuild appends a fresh generated block at the end of the
  body (with a warning) rather than touching any existing text.

### When the markers are ambiguous, MindVault refuses to write

`map rebuild`, `map create`, `mindvault_get_project_map` and the summary-block splice all
classify the marker occurrences on a note before touching it, into one of three cases:

- **None** — the marker string does not appear. Create appends a fresh block; rebuild appends
  one with a warning (as above).
- **Single** — exactly one well-formed start/end pair. This is the normal case: the block is
  rewritten in place and everything outside it is preserved verbatim.
- **Ambiguous** — the marker strings appear more than once, or appear malformed (e.g. a start
  with no matching end, or ends before starts). MindVault **refuses to write**: no block is
  spliced, no snapshot is taken, nothing on the note changes, and it warns you to edit the note
  so the literal marker string appears exactly once as a clean pair.

The refusal exists because there is no safe way to guess *which* occurrence is the real
generated block. Splicing against the wrong one would overwrite or delete human text between
the markers — exactly the outcome the generated-block contract promises never happens. Refusing
is the only choice that cannot lose your writing.

The usual trigger is innocent: a human quotes a marker in prose, or pastes the literal
`start`/`end` line into a code fence while documenting how the block works. Because MindVault
matches the marker as an exact substring, that quoted copy counts as a second occurrence.
**Practical tip:** when you need to mention a marker in the note's human text, break the literal
up so it is not an exact substring match — write it with a zero-width break, split it across a
line, or describe it (`the mindvault-generated start marker`) rather than pasting it whole.

### Idempotent rebuilds

Rebuild is a no-op when nothing changed. MindVault compares the regenerated block against
the current one, ignoring two lines that would otherwise churn on every run: the volatile
`Last Rebuilt` timestamp, and the `Organisation Score` line. The score line is ignored because
it self-references block size — the block contributes to `mapCoverage` and the overall figure,
so materialising the block can nudge the score by a point, which would trigger a fresh rebuild,
which would move it again, forever. If the regenerated block differs from the current one **only**
in the Last Rebuilt timestamp and/or the Organisation Score line, **nothing is written**: no
snapshot, no `updated` bump, no index churn. Running `map rebuild` (or `compile`) repeatedly on
a settled project costs nothing.

### Coexistence with summary blocks

A hub can carry both a map block and a generated **summary** block
(`<!-- mindvault-summary:start -->` / `<!-- mindvault-summary:end -->`, see
[GENERATED_SUMMARIES.md](GENERATED_SUMMARIES.md)). The two use distinct markers and are
spliced independently, so they never collide on the same note.

### Size and token heuristics ignore the block

Large-note flags, summary candidates, low-value detection and token-waste accounting all
measure a note **excluding** its generated block content. A hub carrying a full map block is
therefore never falsely flagged as an oversized note or a summary candidate on account of the
generated text. (The index schema version bumps for this in v0.7.0, so existing indexes
rebuild automatically on first run — no manual step.)

## Migration from `09_Maps`

Before v0.7.0 each project had a separate `09_Maps/<Project> Map.md`. `map create`,
`map rebuild` and `compile` detect a legacy file and handle it safely:

- **No human text outside the markers** — the legacy file is pure generated content, so it
  is archived (snapshot-first, never deleted) once the block lands on the hub.
- **Human text present outside the markers** — MindVault does **not** touch it: it warns and
  leaves the file in place for you to move the human content onto the hub yourself, then
  delete or archive the leftover.

`09_Maps` is no longer a required folder and `init` no longer creates it. Any legacy map
files that remain are excluded from orphan, low-value and audit reports, so an un-migrated
vault reports no new warnings.

## Why one note

The separate map duplicated the hub's goal and non-negotiables, and those copies went stale
between rebuilds. It also doubled every project into two hub-like nodes in Obsidian's graph.
Merging the map into the hub makes the project note the single centre of its cluster and
removes an entire class of staleness — there is now exactly one page per project.

## Usage

- **Humans:** open the project note in Obsidian; the generated block is its live table of
  contents and health readout, and the graph view around the hub shows the project cluster.
- **Agents:** `mindvault_get_project_map` returns just the block content from the hub — the
  cheapest orientation read there is. `mindvault_list_maps` shows which projects have a map
  block (and flags any legacy files). Rebuild after meaningful memory changes (the
  architecture-memory and organisation skills say when).
